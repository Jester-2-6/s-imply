Reconvergent Path Picking & Value Prediction Status Report (20 Nov 2025)
======================================================================

1. Scope and Inputs
- Codebase: s-imply (root). Focused files: src/atpg/reconv_podem.py, src/ml/reconv_path_picker_new.py, src/ml/reconv_ds.py, src/ml/reconv_lib.py, src/ml/train_reconv.py, src/ml/evaluate_reconv.py, src/ml/reverse_simulator.py, src/ml/trainer.py, src/ml/embedding_extractor.py.
- Datasets: Pickled reconvergent structures under data/datasets (plus optional processed shards in data/datasets/reconv_processed). Benchmarks live under data/bench/*.
- Checkpoints: Multiple runs captured in checkpoints/* (anchor_sanity, reconv_constrained, reconv_util_max, reverse simulator runs, etc.), establishing that training scripts have been exercised end-to-end.

2. Reconvergent Path Discovery & Dataset Construction
- Reconvergent pair mining lives in src/atpg/reconv_podem.py. We perform a beam search from multi-fanout sources, rank frontier nodes by branching plus overlap with seed fanouts, and emit every pair of branches that reconverge before a depth cap. This produces exact node-id paths plus metadata (start, reconv node, branch list) without attaching logic assignments.
- The dataset builder (build_dataset + save_dataset) iterates over every .bench file, extracts structural/functional embeddings via EmbeddingExtractor (AIG conversion + mapping), then enumerates reconvergent pairs. Each sample stores the bench path, reconv info, struct_emb tensor, and gate_mapping dict so downstream tooling can map original node ids to embedding rows.
- ReconvergentPathsDataset (src/ml/reconv_ds.py) ingests those pickles and optionally pre-materialized shards. It pads/truncates paths to max_length, injects three logic-value channels (one-hot 0/1/X) when add_logic_value=True, and can pre-write anchor hints directly into the embedding tail. Processed shards (preprocess_to_shards) normalize all samples to consistent shapes and cut I/O overhead by loading tensors in bulk.
- Dataset samples currently lack ground-truth logic values; instead we annotate every node with structural embeddings + default X logic vectors. Preferred anchor values are generated heuristically from gate type (AND/NAND bias to 1, OR/NOR to 0, otherwise prefer_value) to guide training despite missing supervision.

3. Path Selection & Analysis Infrastructure
- src/ml/reconv_path_picker_new.py generalizes dataset handling for path selection experiments. It wraps the pickle dataset as a torch Dataset that lazily loads embeddings per circuit via EmbeddingExtractor, aligns node IDs, and produces padded tensors plus masks ready for downstream models.
- The module defines a PathSelector facade that caches extracted paths per circuit and calls evaluate_path() using a pluggable torch.nn.Module. Today _extract_paths returns an empty list (stub), so automated extraction still depends on reconv_podem outputs; wiring this up is an open item.
- Rich analysis utilities already exist: PathMetrics/PathAnalysisResult capture depth, complexity, convergence, and criticality (compute_* helpers). There is support for clustering, deduplication, adaptive filtering, caching, ensemble scoring (PathScoreAggregator), multi-objective Pareto selection, and hierarchical selection strategies. These indicate a roadmap for ranking and pruning large path sets once the extraction stub is implemented.
- PathTransformer (simple self-attention over padded paths) and helper functions (process_paths, aggregate_path_features, create_path_attention_weights) provide a lightweight baseline for scoring. However, the current implementation performs a single self-attention pass without positional encodings or cross-path interaction, so performance will depend heavily on input embeddings.

4. Multi-Path Value Prediction (Forward Justification)
- Model architecture: MultiPathTransformer (src/ml/reconv_lib.py) accepts B×P×L×D tensors, encodes each path independently via a shared TransformerEncoder, pools the first token to summarize, exchanges information across paths with a second encoder, then broadcasts that global context back to per-node representations before a 2-class linear head predicts logic values. PositionalEncoding is stubbed (pass), so sequence order relies entirely on upstream embeddings.
- Training loop (src/ml/train_reconv.py) favors a REINFORCE-style policy objective instead of cross-entropy because labels are unavailable. Key features:
  * make_dataloaders auto-detects processed shards, keeps dataset tensors on CPU when GPU training, and splits 90/10 train/val.
  * Anchoring: _generate_anchor samples one valid path/node per batch element, determines a compatible output value via gate-type heuristics (_compatible_anchor_value), and optionally writes a one-hot value into the final three embedding dims (_inject_anchor_into_embeddings). Anchor hints can also arrive precomputed from the dataset.
  * policy_loss_and_metrics samples discrete 0/1 actions per node, penalizes gate-level constraint violations using GateType-specific rules, enforces reconvergence agreement by comparing last node predictions across paths, adds anchor reward shaping (alpha-weighted), optionally normalizes rewards, and includes entropy regularization.
  * train_one_epoch / evaluate compute averaged loss, reward proxy, and valid_rate (fraction of samples without violations). Debug metrics (_debug_metrics_from_logits) measure edge accuracy, reconvergence match rate, anchor match, and edges per sample for interpretability when verbose logging is enabled.
  * CLI (build_argparser) exposes batch size, model dims, logic-value toggles, anchor options, and loader performance knobs. Checkpoints save both periodic snapshots and best-by validation reward.
- Evaluate script (src/ml/evaluate_reconv.py) loads MultiPathTransformer and reuses policy_loss_and_metrics, but it imports ReconvergentPathsDataset from src/ml/minimal_reconv_dataset (which does not exist) and still expects embedding_dim argument rather than inferring from data. This mismatch blocks out-of-the-box evaluation and should be corrected to reuse src/ml/reconv_ds.py plus observed dim logic from train_reconv.
- Current checkpoints (e.g., checkpoints/reconv_constrained, reconv_util_max, reconv_rl, anchor_sanity) show that the RL trainer has been exercised under different reward weightings/constraints, but there is no accompanying metrics log in repo—only .pth files.

5. Reverse Simulation / Input Value Prediction Track
- src/ml/reverse_simulator.py implements a transformer that predicts input assignments given functional embeddings of inputs and a desired output vector. It constructs a token sequence containing special learnable markers (output token, desired value tokens, separator) plus projected input embeddings, applies positional encoding, runs a TransformerEncoder, then uses cross-attention from inputs to the desired-output token before a sigmoid head emits per-input probabilities. Predict_with_confidence converts outputs to binary pattern + confidence.
- ReverseSimulatorLoss mixes BCE with optional simulation rewards and a consistency term that aligns confidence with desired outputs. create_model builds a default configuration.
- The trainer stack (src/ml/trainer.py + src/ml/training_env.py) creates CircuitDataset objects by extracting embeddings for every circuit via EmbeddingExtractor, sampling random desired outputs, and—crucially—fabricating random input patterns as “ground truth.” This bootstraps supervised training but does not yet couple to an actual simulator, so value targets remain synthetic. RL hooks (ReverseSimulatorEnv, _update_model_rl) exist but use a simplified update every 10 episodes and rely on env.predict_action wrappers that still need integration with real circuit simulation feedback.

6. Current Assets and Supporting Utilities
- EmbeddingExtractor (src/ml/embedding_extractor.py) standardizes bench→AIG conversion, ensures staging directories are refreshed, and returns both structural and functional embeddings plus gate mappings. This underpins every pipeline discussed above.
- Data directories include data/bench/* suites (ISCAS85/89, ITC99, RCCG), arbitrary micro tests, and staging patterns. Pretrained weights for reverse simulation reside under weights/ (best_model.pth, reverse_simulator_weights.pth, etc.).
- Utility code (src/util/io.py, src/util/struct.py) provides parse_bench_file and Gate/GateType definitions needed throughout. These appear stable and reused broadly.

7. Key Gaps and Suggested Next Steps
- Finish reconv path extraction in PathSelector._extract_paths and connect it to the reconv_podem dataset so the rich ranking/filtering toolkit can be exercised on real paths instead of empty lists.
- Implement the missing PositionalEncoding in src/ml/reconv_lib.py or perform absolute/relative position injection earlier to stabilize long-path modeling.
- Record or derive supervised labels (e.g., via constrained logic simulation or PODEM) to enable cross-entropy training alongside the current RL reward, or at least persist reward/accuracy curves per checkpoint for reproducibility.
- Fix src/ml/evaluate_reconv.py to import ReconvergentPathsDataset from src/ml/reconv_ds.py, infer embedding dimension from data, and optionally load anchor hints to mirror training.
- Replace random input patterns in CircuitDataset with patterns generated by actual simulation/PODEM to make reverse_simulator training meaningful; tighten RL loop by wiring ReverseSimulatorEnv.step to real circuit feedback and using mini-batch policy/value updates.
- Document checkpoint provenance (hyperparameters, dataset snapshots) in docs/ or README to make current assets actionable, and consider exporting aggregated metrics from _debug_metrics_from_logits for later analysis.
